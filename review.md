## 3. 🔴 ТОП-5 критических проблем

### #1 [P0, блокер] Уход за эпическим драконом полностью сломан (TypeError)

**Файл:** `bot/handlers/epic_care.py:92`
**Симптом:** При нажатии «Норма» или «Штраф(x2)» на стадии ухода — вызов падает, прогресс эпика невозможен.

`fsm.py:95` определяет функцию **без** параметра `suffix`:
```python
def epic_care_state(stage_id: int) -> str:        # ← один аргумент
    return f"epic_care_{stage_id}"
```

Но `epic_care.py:92` вызывает её **с двумя**:
```python
def handle_care_mode(user, mode, db, send_message):
    ...
    user.state = epic_care_state(care.stage_id, mode)   # ← TypeError!
```

При этом `is_epic_care_waiting` (`fsm.py:103`) ожидает суффиксы `_norm/_x2`, т.е. контракт подразумевает суффикс, но builder его не принимает. Асимметрия с `grow_state`/`legend_state`/`epic_egg_state`, у которых `suffix` есть.

**Эффект по цепочке:** пользователь видит действие ухода → нажимает «Норма» → `TypeError` → состояние не меняется → `is_epic_care_waiting` всегда False → `handle_care_message` (`main.py:289`) **никогда не вызывается**. Вся Фаза 5/6 (цикл ухода, стадии, финал, характер, мудлеты) недостижима.

**Тесты это пропустили:** `test_epic_handler.py` покрывает только имя и рестарт, **не покрывает режим ухода через main.py-диспетчер**.

**Исправление** — добавить `suffix` в `fsm.py` для единообразия:
```python
def epic_care_state(stage_id: int, suffix: str = "") -> str:
    base = f"epic_care_{stage_id}"
    return f"{base}_{suffix}" if suffix else base
```
И добавить тест, проходящий путь «show_care_action → handle_care_mode('norm') → handle_care_message('вышито N')» через диспетчер `main.py`.

### #2 [P1] Контракт «подозрительные отчёты игрока» нарушен — фильтрация на фронте

**План (Фаза 1.5):** «Карточка «⚠ Подозрительные отчёты» **в детали игрока**».

**Реализация:** `GET /admin/users/{vk_id}` (`admin.py:454`) **не возвращает** отчёты. Фронт `UsersList.tsx:34` тянет ВСЕ отчёты и фильтрует на клиенте:
```typescript
const sr = await client.get('/admin/suspicious');          // весь список
setSuspicious(sr.data.filter((s) => s.user_id === id));    // фильтр на фронте
```

Это N+1 по данным (грузится до 500 отчётов при открытии любого игрока) и рассогласование контракта. **Исправление:** добавить в `get_user_detail` поле `suspicious_reports` (фильтр по `user_id`), либо эндпоинт `GET /admin/users/{vk_id}/suspicious`.

- Этап 3: **1–2 недели** (опционально).

**Главный совет:** текущие тесты покрывают сервисы изолированно, но **не интегрируют диспетчер `main.py`**. Добавьте 2-3 end-to-end теста (сообщение «вышито N» в состоянии `epic_care_X_norm` → проверка смены состояния и БД) — это поймало бы баг #1 ещё на CI.
