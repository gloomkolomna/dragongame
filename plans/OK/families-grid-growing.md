# Семейства, альбомы и выращивание — дизайн

> Дата: 2026-07-02. Охватывает: модели данных, API, Mini App, Bot FSM.

## 1. Семейства драконов

### 1.1 Модель

Новая таблица `families`:

| Колонка | Тип | Описание |
|---|---|---|
| `id` | INTEGER PK | |
| `name` | VARCHAR NOT NULL | Название семейства (напр. «Огненные») |
| `description` | TEXT | Краткое описание |
| `sort_order` | INTEGER DEFAULT 0 | Порядок отображения в интерфейсе |

В таблицу `dragons` добавляется:

| Колонка | Тип | Описание |
|---|---|---|
| `family_id` | INTEGER FK → families.id, ON DELETE SET NULL | К какому семейству относится |

**Правила:**
- Каждый дракон относится ровно к одному семейству (или ни к какому — `NULL`)
- Семейство можно создать/удалить через админку
- При удалении семейства драконы не удаляются (SET NULL), их можно перепривязать

### 1.2 Админка: управление семействами

Новая страница `/admin/families`:
- Таблица: Название | Описание | Кол-во драконов | Порядок
- Кнопки: Создать / Редактировать / Удалить
- Форма: название, описание, порядок

В форме редактирования дракона (`/admin/dragons/:id/edit`) — выпадающий список выбора семейства.

## 2. Альбомы (сетки) на каждое семейство

### 2.1 Модель

Меняется таблица `collection_grid`:

| Колонка | Тип | Изменение |
|---|---|---|
| `family_id` | INTEGER FK → families.id, NOT NULL | **НОВОЕ** |
| `cell_x` | INTEGER | |
| `cell_y` | INTEGER | |
| `dragon_id` | INTEGER FK → dragons.id, UNIQUE | |

Уникальность: `UNIQUE(family_id, cell_x, cell_y)` + `UNIQUE(dragon_id)` (дракон не может быть в двух ячейках).

**Логика:**
- У каждого семейства — своя отдельная сетка (альбом)
- Администратор создаёт/редактирует сетку для каждого семейства отдельно
- Размер сетки: произвольный (M×N), задаётся админом
- Одна ячейка = один дракон (или пустая)

### 2.2 Админка: управление альбомами

Страница `/admin/grid` перерабатывается:
- Сверху — выбор семейства (выпадающий список)
- При выборе семейства показывается его сетка
- Кнопки: «Создать сетку», «Изменить размер» (+/- строки/столбцы)
- Ячейки: выбор дракона из выпадающего списка (только драконы этого семейства, не размещённые в других ячейках)

### 2.3 API для Mini App

Текущий `GET /api/collection/{vk_id}` заменяется на:

```
GET /api/collection/{vk_id}?family_id=1
```

Ответ содержит сетку конкретного семейства + статус каждой ячейки для данного пользователя.

```
GET /api/collection/{vk_id}/families
```

Возвращает список семейств (id, name, sort_order, total_dragons, collected_by_user).

## 3. Механика выращивания — Mini App

### 3.1 Состояния ячейки

Каждая ячейка в альбоме для конкретного пользователя находится в одном из трёх состояний:

| Состояние | Условие | Что видно в ячейке |
|---|---|---|
| **locked** | Пользователь не активировал PIN этого дракона | Серая ячейка, знак `?` или замок |
| **growing** | PIN активирован, шаги не все пройдены | **Яйцо** (silhouette) + индикатор прогресса |
| **completed** | Все шаги пройдены | **Взрослый дракон** (image) + имя |

### 3.2 Визуализация

**Growing (яйцо + прогресс):**
```
┌──────────────┐
│              │
│    🥚        │  ← silhouette_path (силуэт яйца)
│              │
│   64%        │  ← progress_pct
│   ████░░     │  ← мини-прогресс-бар
└──────────────┘
```

**Completed (взрослый дракон):**
```
┌──────────────┐
│  🐉          │  ← image_path (полное изображение)
│  Пламекрыл   │  ← name
└──────────────┘
```

При тапе на ячейку:
- **growing** → страница `/dragon/:id` (показывает: яйцо, текущий шаг, список шагов с ✅/→)
- **completed** → страница `/dragon/:id` (показывает: изображение, имя, редкость, описание, все шаги ✅)
- **locked** → не кликабельна

### 3.3 Навигация в Mini App

```
┌─────────────────────────────┐
│  🏠 Моя коллекция           │  ← заголовок
├─────────────────────────────┤
│  Вкладки семейств:           │
│  [Огненные] [Ледяные] [Все] │  ← горизонтальный скролл
├─────────────────────────────┤
│                             │
│   Сетка текущего семейства   │  ← ячейки locked/growing/completed
│   (4×4 или другой размер)   │
│                             │
├─────────────────────────────┤
│  Собрано: 3 из 12           │  ← статистика
└─────────────────────────────┘
```

- Вкладка «Все» показывает объединённую сетку (или список всех семейств с мини-превью)
- По умолчанию открывается первое семейство по `sort_order`

## 4. Бот — полная FSM

### 4.1 Обзор состояний

```
                 ┌──────────────────────────┐
                 │         IDLE             │
                 │  (бот ждёт команду или    │
                 │   первый запуск)          │
                 └─────┬──────────┬─────────┘
                       │          │
              /start   │          │ любой текст (не команда)
                       ▼          ▼
              ┌──────────┐  ┌──────────────┐
              │AWAIT_PIN │  │ подсказка:    │
              │ждёт ввода │  │ "напиши /pin │
              │PIN-кода   │  │  чтобы начать│
              └────┬──────┘  │ выращивание" │
                   │         └──────────────┘
       ввёл "4729" │ (верный PIN)
                   ▼
         ┌─────────────────┐
         │  GROWING        │
         │  (шаг 1..N)     │◄──────────────────────┐
         │                 │                       │
         │  Бот описывает   │  фото + "вышито"      │
         │  задание шага    │──────► проверка ──────┘
         │                 │       (неверно: повторить)
         └────────┬────────┘
                  │ все шаги пройдены
                  ▼
         ┌─────────────────┐
         │  COMPLETED      │
         │  дракон выращен! │
         │  показать имя +  │
         │  картинку         │
         └────────┬────────┘
                  │ автоматически через ~5 сек
                  ▼
         ┌─────────────────┐
         │  IDLE           │
         │  (готов к новому │
         │   выращиванию)   │
         └─────────────────┘
```

### 4.2 Таблица состояний

| Состояние | `user.state` | Описание |
|---|---|---|
| IDLE | `idle` | Бездействие. Ждёт команду `/pin` или `/start`. |
| AWAIT_PIN | `await_pin` | Ждёт ввода 4-значного PIN-кода. |
| GROW_STEP | `grow_step_{N}` | Выполняет шаг N. Ждёт фото + ключевое слово. |
| COMPLETED | `completed` | Все шаги пройдены. Показ результата. Переход в IDLE. |

В БД состояние хранится как `grow_step_1`, `grow_step_2`, ..., `grow_step_N`, `idle`, `await_pin`, `completed`.

### 4.3 Команды

| Команда | Где работает | Действие |
|---|---|---|
| `/start` | Любое состояние | Приветствие. Если есть незавершённый дракон → продолжить. Иначе → предложить `/pin`. |
| `/pin` | IDLE, GROW_STEP | **v1:** только из IDLE. Переход в AWAIT_PIN. **v2:** можно из любого состояния — если уже есть активный дракон, переключает на нового (старый «замораживается»). |
| `4729` (PIN) | AWAIT_PIN | Проверка PIN. Верный → старт выращивания (шаг 1). Неверный → «PIN не найден, попробуй ещё раз». |
| `/help` | Любое | Справка по командам. |
| `/status` | Любое | Показывает текущий прогресс: имя (если известно), шаг N из M, описание задания. |
| `/garden` | Любое (v2) | Список всех активных драконов с прогрессом. Позволяет переключиться на любого. |
| `/skip` | GROW_STEP | (Только для админов/теста) Пропустить текущий шаг. |
| `/reset` | Любое | Сбросить текущее выращивание. Подтверждение: «точно? напиши ДА». |

### 4.4 Процесс шага (GROW_STEP_N)

```
Бот:  «Шаг 2 из 5: Вышей хвост дракона красной нитью.
       Когда закончишь — сфотографируй и отправь фото
       с подписью "вышито".»

Пользователь: [отправляет фото]
Пользователь: вышито

Бот:  «✅ Шаг 2 выполнен! Твой дракон подрос.
       ████░░░░░░ 40%»

       [если есть след. шаг]
       «Следующий шаг: Шаг 3 из 5 — ...»
       [переход в GROW_STEP_3]

       [если последний шаг]
       «🎉 Поздравляю! Ты вырастил дракона!
        ⭐ Пламекрыл ⭐
        Редкость: легендарный
        [изображение дракона]
        
        Загляни в мини-приложение, чтобы увидеть
        его в своей коллекции!»
       [переход в COMPLETED → IDLE]
```

### 4.5 Валидация фото

- Пользователь отправляет фото (бот сохраняет photo_id ВКонтакте)
- Пользователь отправляет сообщение с ключевым словом `"вышито"`
- Бот проверяет: было ли фото в этом же сообщении или предыдущем (в течение 5 минут)
- Если фото + ключевое слово → шаг засчитан
- Если только текст без фото → «Сначала отправь фото, потом напиши "вышито"»
- Если только фото без текста → «Отправь фото и напиши "вышито"»

### 4.6 Несколько активных драконов

**Можно, но через переключение, не параллельно.** Бот ВК — линейный чат: в каждый момент пользователь работает с одним драконом. Но в БД может быть несколько драконов в статусе «growing», а пользователь переключается между ними.

#### Как устроено переключение

Модель данных уже поддерживает несколько активных драконов:
- `UserProgress` — одна запись на каждую пару (user, dragon, step) — прогресс не теряется
- `UserDragon` — запись создаётся при активации PIN, `completed_at=NULL` пока дракон растёт
- `User.current_dragon_id` — указывает, с каким драконом пользователь работает **сейчас**

При активации нового PIN, когда текущий дракон ещё не завершён:
- Текущий дракон «замораживается» — его прогресс сохраняется в `UserProgress`
- `User.current_dragon_id` переключается на нового
- Пользователь начинает выращивать нового дракона с шага 1

#### Команда `/garden`

```
User: /garden
Bot:  🪴 Твой сад:
      1. 🥚 Ледяное яйцо — шаг 2 из 4 (50%)  ← сейчас
      2. 🥚 Огненное яйцо — шаг 1 из 3 (33%)
      Напиши номер, чтобы переключиться.

User: 2
Bot:  ▸ Переключился на «Огненное яйцо».
      Шаг 1 из 3: Вышей контур дракона золотой нитью.
      [ждёт фото + "вышито"]
```

После переключения все сообщения (фото, «вышито», `/status`) относятся к выбранному дракону. FSM не меняется — просто `current_dragon_id` обновляется, состояние остаётся `grow_step_N`.

#### Ограничения ВК

| Можно | Нельзя |
|---|---|
| `/garden` — список активных драконов + выбор | Два параллельных диалога в одном чате |
| Одно фото = один шаг одного дракона | Отправить фото «для всех активных драконов сразу» |
| Явное переключение через `/garden` или кнопки | Авто-определение «это фото для какого дракона?» |
| Несколько драконов в «замороженном» состоянии | Одновременный приём заданий для разных драконов |

#### Сценарий с тремя драконами

```
1. Игрок купил 3 яйца
2. /pin → 4729 → начал «Ледяное» (шаг 1/4)
3. Сделал шаг 1 (фото + "вышито") → шаг 2/4
4. /pin → 8391 → бот: «У тебя уже есть активный дракон.
   Переключить на нового? (текущий прогресс сохранится)»
   Игрок: да → начал «Огненное» (шаг 1/3)
5. Сделал шаг 1 → /garden → 1 → вернулся к «Ледяному» (шаг 2/4)
6. Сделал шаги 2,3,4 → «Ледяное» completed! → показан дракон
7. /garden → 1 (теперь это «Огненное») → продолжил
```

#### План внедрения

| Версия | Поведение |
|---|---|
| **v1** (старт) | Один дракон за раз. Новый PIN = сброс текущего (с подтверждением). Прогресс сохраняется в БД, но активен только один. |
| **v2** | `/garden` для переключения. Можно иметь несколько активных драконов. Без подтверждения при `/pin` — просто переключает. |
| **v3** (опционально) | Inline-клавиатура после каждого шага: [Продолжить этого] [Выбрать другого]. Визуальные кнопки вместо текстовых команд. |

## 5. Взаимодействие бота и Mini App

### 5.1 Сценарий игрока

```
1. Игрок покупает физическое яйцо
2. Внутри яйца — листок с PIN-кодом (4 цифры) и стикер дракона
3. Игрок открывает VK, находит бота сообщества
4. Пишет /pin → вводит PIN-код
5. Бот подтверждает: «Ты начал выращивать яйцо "Ледяное"!»
6. Бот выдаёт первое задание (шаг 1)
7. Игрок вышивает → фотографирует → отправляет боту с подписью "вышито"
8. Бот подтверждает шаг, показывает прогресс, выдаёт следующее задание
9. Шаги 2..N повторяются
10. После последнего шага — дракон «вылупляется», бот показывает имя и картинку
11. Игрок открывает Mini App → вкладка семейства → ячейка показывает дракона (вместо яйца)
12. Игрок собирает коллекцию: покупает новые яйца → повторяет цикл
```

### 5.2 Синхронизация состояний

- Бот пишет в БД: `User.state`, `UserProgress`, `UserDragon`
- Mini App читает из API: `GET /collection/{vk_id}?family_id=N`
- Мини-апп показывает актуальное состояние без вебсокетов (polling при открытии)
- Бот после завершения шага может отправить уведомление в Mini App (через VK Bridge `VKWebAppSendPayload` или просто — игрок сам обновит страницу)

```sql
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name VARCHAR NOT NULL,

```

### 6.2 Изменения существующих таблиц

```sql
-- dragons
ALTER TABLE dragons ADD COLUMN family_id INTEGER REFERENCES families(id) ON DELETE SET NULL;

-- collection_grid
ALTER TABLE collection_grid ADD COLUMN family_id INTEGER NOT NULL DEFAULT 0;
```
