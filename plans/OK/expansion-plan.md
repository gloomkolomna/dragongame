# Поэтапный план реализации расширения «Бестиарий драконьих легенд»

## Вердикт
Идея **реализуема в текущей кодовой базе**. ~80% переиспользует готовую механику «PIN → шаги вышивки → дракон». Ядро (яйцо → шаги с крестиками/нормой/таймаутом → взрослый дракон) остаётся основой и **расширяется**, а не переписывается. Эпический тоже растёт из яйца по шагам, а после вылупления входит в слой «ухода» по стадиям.

**Принцип «данные, а не код»:** стадии эпического, их действия, таймауты, товары стадий — данные в БД, настраиваются через админку. Код = универсальный движок стадий. Число стадий не фиксировано.

Разбито на **8 фаз (0–7)** по зависимостям. Каждая фаза самодостаточна, тестируема и шипится отдельно.

## Зафиксированные решения
1. **Копилка** = ВСЕ крестики из каждого «вышито N» по всем яйцам.
2. **Наград за редкость нет.** Редкость влияет только на контент (легендарный → легенда; эпический → стадии). Сокровищ нет.
3. **Анти-чит 5×:** число > нормы в 5 раз → крестики зачисляются, отчёт помечается подозрительным; игроку — уведомление о проверке; админ — очередь + VK-дублирование, корректирует копилку вручную.
4. **Ссылка на чат игрока:** `https://vk.com/gim239999455/convo/{vk_id}`.
5. **Подозрительные отчёты во frontend:** только список + кнопка «Перейти в чат с игроком» (open_link). Inline-модерации/закрытия нет.
6. **Магазин:** бот — просмотр **и** покупка; **мини-апп — ТОЛЬКО просмотр (read-only)**. Покупка идёт только через бот-логику. API магазина — только чтение.
7. **Товар → стадии автоматически.** Товар доступен на стадии тогда, когда он назначен в действии ухода этой стадии. Привязка (`stage_shop_items`) синхронизируется автоматически из товаров действий; вручную не настраивается. Категория товара не используется.
8. **Каждый товар формирует характер.** У товара есть необязательное поле **«Влияние на характер»** (`character_effect`). Характер выращенного дракона складывается из купленных за выращивание товаров.
9. **Покупка один раз за этап.** Игрок покупает товар для стадии один раз, а не каждый цикл (`user_inventory` — уникальная пара user/item + количество).
10. **Уход за эпическим:** таймаут **по умолчанию 24ч, настраиваемый** на действии. Scheduler шлёт просьбу **1 раз**, новый отсчёт — только после выполнения.
11. **Крестики в стадиях по общим правилам.** Каждое действие ухода имеет `crosses_norm`; засчитывается по норме/штрафу ×2, как обычные шаги.
12. **Действие ухода → несколько товаров.** Действие может требовать несколько товаров (`epic_action_items`, many-to-many).
13. **Стадии выращивания настраиваются админом** (название, описание, 2 картинки — начало/финал стадии, число циклов, таймаут, действия). Не хардкод.
14. **Блоки выбора убраны.** Отдельного движка «выборов на стадии» нет. Характер задаётся через покупки (п. 8). Имя даётся отдельным шагом после вылупления.
15. **Эпический растёт из яйца по шагам** (`DragonStep`, `phase=0`). После вылупления бот предлагает **дать имя**, затем начинаются стадии ухода.
16. **Эпический маркируется флагом `is_epic`.** `rarity` остаётся 1–3, rarity 4 не вводится. Форма эпического вида без редкости/семейства/PIN — только имя, тип яйца, описание, **одна картинка (яйцо)** и шаги выращивания яйца.
17. **Видов эпических много, этапы выращивания общие** (создаются и настраиваются один раз, без привязки к виду).
18. **Один эпический за раз** у игрока (отдельный слот, не смешивается с обычными).
19. **Первое появление эпического** — авто, один раз, случайный вид из пула после первого завершённого яйца.
20. **После финала** игрок выбирает: такого же заново ИЛИ нового случайного. Без запроса новый не выдаётся.
21. **Финал** — завершение последней стадии (макс. `stage_number`), не хардкод-номер.
22. **Эпический + этапы** — отдельная вкладка админки (создание вида только там).
23. **Легенды в админке** — отрывки-стадии (`DragonStep phase=1`) с фото к каждому + обложка легенды.
24. **Книга из легенды** — остаётся (товар магазина).
25. **Легендарный после вылупления «висит» активным** и ждёт режима легенды (Фаза 3).
26. **Действия ухода** различаются по id/названию/товарам/норме/порядку. Порядок меняется перетаскиванием; технического кода-типа у действия нет.

## Сквозные архитектурные решения
- **Время:** строковый подход (`datetime.fromisoformat` + `next_step_available_at`).
- **Миграции:** несколько ревизий (Фаза 0 — набор ревизий; далее по необходимости). Защитные idempotent-миграции для дрейфа dev-БД.
- **Импорт моделей:** lazy-import внутри функций (`bot/services/*.py`).
- **Кнопки ВК:** ≤5 в строке / ≤10 строк. Динамические клавиатуры магазина — с пагинацией.
- **FSM:** `User.state` хранит только грубый режим; точный прогресс — в данных эпического/легенды.
- **stage_key:** `epic:{stage_number}` (egg = `epic:egg`). Используется магазином.
- **Идемпотентность:** эпический триггер, выдача мудлетов, открытие фрагментов легенды — без повторов.
- **Обратная совместимость:** новые таблицы/поля с дефолтами.
- **Тесты:** после каждой фазы `api\venv\Scripts\python.exe -m pytest api/tests bot/tests -v`; фазы с фронтом — `cd frontend; npx vitest run`. Обновлять счётчик в AGENTS.md.
- **Исправить по ходу** (расхождения с AGENTS.md): скрытие будущих шагов в коллекции; `complete_dragon → COMPLETED`.
- **Анти-чит порог (5×)** вынести в константу/конфиг.

## Фаза 2 — Магазин (бот: покупка; мини-апп: только просмотр) 🚧 БОТ+API ГОТОВО

## Фаза 5 — Эпический: цикл ежедневного ухода ⚠ самая сложная ✅ ГОТОВО
**Зависимости:** 2, 4. **Сложность:** 🔴 высокая.

### Поведение
- Стадия = упорядоченный цикл действий (кормить→пелёнка→купать→убаюкать…). Цикл повторяется `cycles_count` раз, затем переход стадии. Стадии общие для всех видов.
- Каждое действие = задание с крестиками по общим правилам (`crosses_norm`, норма/штраф ×2).
- Реальный таймаут: после действия → `next_action_at = now + care_timeout_*` (24ч дефолт). Scheduler шлёт напоминание **1 раз**; новый отсчёт — только после выполнения.
- Действие требует **товары из инвентаря** (`epic_action_items`, может быть несколько) → расход. Товары стадии = из `stage_shop_items` (авто из действий).
- **Покупка один раз за этап** — купленный товар остаётся в инвентаре на стадии.
- Прогресс эпического заблокирован, пока действие не выполнено. Локально, не влияет на других.

### Код/хендлеры
- `bot/fsm.py`: `epic_care_<stage_id>`; текущее действие в `state_data`.
- `bot/keyboard.py`: `care_actions_keyboard`.
- `bot/handlers/epic.py`: уход, проверка инвентаря (все товары действия), advance.
- `bot/services/epic_service.py`: `advance_action`, `advance_stage`, `set_care_timeout`, `get_care_remaining`.
- `bot/scheduler.py`: `_check_care_due` (интервал из действия).

### Детальный чеклист
- [x] `bot/fsm.py`: `epic_care_<stage_id>` (+`_norm/_x2`), хелперы `is_epic_care`/`is_epic_care_waiting`
- [x] `epic_service.advance_care` — действие → цикл → стадия → финал (события `next_action`/`cycle_done`/`stage_up`/`finale`)
- [x] `epic_service.set_care_timeout` / `get_care_remaining` — `next_action_at`, `care_notified`
- [x] Каждое действие = задание с крестиками (`crosses_norm`, норма/штраф ×2, зачисление в копилку)

## Фаза 6 — Финал стадий + мудлеты + сбор характера ✅ ГОТОВО

### Финал
- Последняя стадия (макс. `stage_number`): дракон улетает и подкидывает нового малыша → игрок выбирает «такого же заново» или «нового случайного». Дракон уходит из активного списка, остаётся в бестиарии с мудлетами и характером.

### Код/хендлеры

## Риски и точки внимания
- **FSM:** прогресс эпического — в данных эпического; `User.state` — грубый режим.
- **Лимит кнопок ВК:** пагинация магазина, ≤5/строка.
- **Идемпотентность:** эпический триггер, мудлеты, фрагменты легенды.
- **Анти-чит порог (5×):** в конфиг/константу.
- **Данные, а не код:** стадии/действия/таймауты/товары стадий — через админку.
- **Таймаут ухода:** 24ч дефолт, редактируется; scheduler читает из действия.
- **Легендарный в ожидании легенды:** между Фазой 1 и 3 активен, но бездействует.
- **Пул эпических = 1 вид:** «новый случайный» = тот же + сообщение.
- **Обратная совместимость + дрейф dev-БД:** защитные idempotent-миграции.
